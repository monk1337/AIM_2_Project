# Phase 1, Off-the-shelf Surgical Hand-Pose Benchmark

**Authors:** Ankit Pal, Yuchang Su (HMS, BMI)
**Date:** 2026-04-25
**Status:** Phase 1 complete (4 of 7 models). Phase 2 fine-tuning pending.

> First quantitative benchmark of MANO-based hand-pose estimators on **real surgical OR data** (Aria val n=2,333) alongside synthetic surgical data (POV-Surgery test n=28,802), with per-video breakdown and dual-GT eval.

---

## 1. Project goal

Mid-term thesis: state-of-the-art hand pose estimators (WiLoR, HaMeR) fail on surgical hand data, ~11 mm PA-MPJPE on synthetic POV-Surgery, qualitatively bad on real OR with no quantitative number.

Final-term goal: close the synth→real gap by (a) quantifying it across multiple architectures and (b) fine-tuning the best-suited base model on POV-Surgery, evaluating transfer to real OR.

## 2. Datasets

### POV-Surgery test (synthetic, in-distribution-ish for HaMeR-family)
- HuggingFace: `harvardsil/pov-surgery` split=test
- **n = 28,802** hand instances across 13 sequences (3 tools: diskplacer, friem, scalpel)
- All `hand_side='right'` (single-hand recordings)
- Image: 1920×1080 landscape pinhole RGB
- Per-row fields: `image`, `joints_3d` (21×3, MANO order, real synthetic GT), `joints_2d` (21×2), `vertices` (778×3), `intrinsics` (3×3 pinhole), MANO params

### Aria val v2-clean (real OR egocentric, headline benchmark)
- HuggingFace: `harvardsil/aria-surgical-hand-pose` split=validation, sequences PR81-85
- Raw rows: 3,259 → filter `has_eval_gt=True` → 3,190 → apply `aaditya/phase0-artifacts/reject_keys_all_20260419.json` (519 reject + skip keys) → **n = 2,333** ("aria_val_v2_clean")
- Image: 1408×1408 fisheye RGB (display frame; **`joints_2d` is in raw sensor frame and needs CCW90 to match**)
- **Two GTs per row:**
  - **Aria-MPS native** (`eval_joints_3d`, 21×3 in **MPS-native landmark order**, needs remap to OpenPose). This is the **primary fair eval target.**
  - **HaMeRSAM pseudo** (`joints_3d`, `joints_2d`, `vertices`, MANO params, all in MANO order, generated by HaMeR + SAM running on the frame, **circular for HaMeR-family models**)
- Sidecar: `aaditya/phase0-artifacts/mps_v2_val_20260419.json` provides pre-projected MPS 2D in display frame (`j2d_v2_disp`), used for P2D metric vs MPS-GT.

## 4. Eval pipeline (canonical lab-comparable)

### 4.1 Joint orders + permutations

```
Aria MPS native (21 slots, palm center at idx 20) ─ARIA_MPS_TO_OPENPOSE→ OpenPose (21, OP[1] dummy)
                                                                              │
                                       OP_VALID = [0, 2..20] (20-joint subset; OP[1] = THUMB_CMC unfilled)
                                                                              ↓
MANO standard (HaMeR/WiLoR/smplx)  ─MANO_TO_OPENPOSE [0,13,14,15,16,1,2,3,17,4,5,6,18,10,11,12,19,7,8,9,20]→ OP
HandOccNet/MGFM internal           ─already in OP order (no perm needed)
```

`MANO_TO_OPENPOSE` and `ARIA_MPS_TO_OPENPOSE` are exactly as defined in the lab's `eval/phase0/phase0/config.py`.

### 4.2 Bbox derivation (canonical)

```python
def derive_bbox_from_joints2d(joints_2d, padding=1.5, img_wh, size_min=260, size_max=1200):
    cx, cy = mean of (min, max)
    side = padding * max(x_range, y_range)        # 1.5× max axis-range
    side = clamp(side, [260, 1200])               # display-frame px clamp
    return [x, y, w, h]                           # square crop, clipped to image
```

**For viz only:** project_via_centroid uses raw `joints_2d.mean()` as anchor (not bbox center), handles cases where joints extend beyond image bounds.

### 4.3 Per-model crop-regress

All 4 models run in **crop-regress mode** (GT bbox derived from native_joints_2d after CCW90 for Aria). YOLO detectors bypassed because they fail on gloves/fisheye (midterm finding).

| Model | Input | Output (3D root-aligned, OP/MANO order) |
|---|---|---|
| WiLoR | 256×256 patch (with skimage gaussian anti-alias if downsampled) | `pred_keypoints_3d` (MANO), `pred_vertices`, `pred_cam` (s,tx,ty) |
| HaMeR | 256×256 ImageNet-normalized patch | same shape |
| HandOccNet | 256×256 [0,1] patch | `joints_coord_cam` (OP, m), `mesh_coord_cam` |
| MGFM | 224×224 ImageNet-normalized patch | `pred_3d_joints` (OP), `pred_vertices` |

For left-hand inputs, image is horizontally flipped (right-hand model trained), then output is un-flipped on x-axis.

### 4.4 Metrics (canonical naming)

| Name | Definition | Units |
|---|---|---|
| `mpjpe_mm` | root-aligned MPJPE (subtract wrist, then `‖·‖₂.mean × 1000`) | mm |
| `mpjpe_abs_mm` | absolute-frame MPJPE (no alignment), **N/A for HaMeR/WiLoR** (root-aligned by construction) | mm |
| `pa_mpjpe_mm` | Procrustes-aligned MPJPE (frame-invariant) | mm |
| `pve_mm` | root-aligned per-vertex error | mm |
| `pa_pve_mm` | Procrustes-aligned per-vertex error | mm |
| `p2d_px` | 2D reprojection error in image pixels | px |
| `root_err_mm` | wrist-only absolute offset, **N/A for root-aligned models** | mm |
| `mrrpe_mm` | Right→Left wrist offset error on both-hand frames, **N/A for root-aligned models** | mm |

Suffix `_v2` = computed on OP_VALID 20-joint subset (vs MPS-GT, since OP[1] is dummy).

### 4.5 Dual-GT eval (Aria val only)

For each row:
- **vs Aria-MPS native** (PRIMARY, fair): convert `eval_joints_3d` MPS-native → OP via `aria_mps_to_op()`, then convert pred MANO→OP via `mano_to_op()`, compare on `OP_VALID` 20 joints. P2D uses sidecar `j2d_v2_disp` (already in display frame).
- **vs HaMeRSAM pseudo** (SECONDARY, **circular for HaMeR/WiLoR**): direct MANO-order comparison on full 21 joints. P2D uses stored `joints_2d` after **CCW90** transform. PVE / PA-PVE only available against HaMeRSAM mesh.

For OP-output models (HandOccNet, MGFM): comparing to MANO-order HaMeRSAM requires `mano_to_op(hsam_3d)` → OP-vs-OP comparison.

### 4.6 Procrustes alignment formula

```
SVD of (P − μ_P).T @ (T − μ_T) = U Σ V.T
R = V · diag(1, 1, sign(det(V·U.T))) · U.T
s = Σ_diag · sign / ‖P − μ_P‖²
t = μ_T − s · R · μ_P
P_aligned = (s · R · P.T).T + t
```

| Model | Aria-MPS | HSAM | gap |
|---|---|---|---|

> **Story-defining finding:** the choice of pseudo-label GT determines which model "wins". On the only fair eval (Aria-MPS native), HO3D-trained models are ~3× better than HaMeR-family on real surgical OR. HSAM pseudo-label would have inverted the conclusion.

## 8. Visualization pipeline (`viz_export.py`)

### 8.1 Two overlay layers per model (Option C)

For each hand instance:
- **`<Model>-shape-{R,L}`** (light color): pred_3d (root-aligned, OP order) → bbox-centroid weak-perspective. Shows skeleton SHAPE pinned at GT hand center. Honest: uses GT info to position; for shape comparison only.
- **`<Model>-pos-{R,L}`** (dark color): actual model 2D in image frame.
  - WiLoR/HaMeR: full perspective_projection through model's own pred_cam + scaled_focal.
  - HandOccNet/MGFM: not generated (their 3D coords don't compose cleanly with GT-wrist + K projection, scale issues).

### 8.2 GT overlays
- **GT-HSAM-{R,L}**: native_joints_2d (CCW90 for Aria, as-is for POV).
- **GT-MPS** (Aria sidecar `j2d_v2_disp`): **dropped from viz**, sidecar uses opposite hand_side convention vs HSAM in the dataset, so MPS-R lands on the visually-other hand. Eval metrics are still fine (per-row eval_joints_3d in parquet has consistent labeling).

### 8.3 Sample selection

- Aria val: 20 frames per PR × 5 PRs = 100 frames stratified on frame_id (np.linspace).
- POV test: 8 frames per sequence × 13 sequences = 104 frames stratified.

## 10. Recommendation for Phase 2 (fine-tuning)

Pick **2 complementary base models** for FT on POV-Surgery train (~70K frames):

1. **WiLoR** (primary)
   - Best off-shelf POV (10.66 mm), worst on Aria (41.63 mm) → biggest headroom
   - Clean codebase, MPS-compatible, easy to fine-tune
   - Tests: "Can we close the synth→real gap by FT-ing on synthetic surgical?"

2. **HandOccNet** (secondary)
   - Best off-shelf Aria (13.88 mm), worst on POV (39.09 mm)
   - HO3D-pretrained (real hand+object), occlusion-aware
   - Tests: "Does adding POV synthetic FT push beyond HO3D's 13.88 mm, or does it hurt?"

Outcome predictions (story-meaningful):
- (a) Both improve → "FT on POV helps regardless of base"
- (b) WiLoR improves a lot, HandOccNet flat or worse → "Pre-training distribution dominates"
- (c) WiLoR converges to HandOccNet's level → "Different paths, same destination, architecture is secondary"

Skip HaMeR (too similar to WiLoR) and Mesh Graphormer (codebase fragile, marginal gain).

### 2×2 PR matrix to ship

|  | Off-shelf | + POV-Surgery FT |
|---|---|---|
| WiLoR (FreiHAND base) | 41.63 mm Aria | TBD (target ~20 mm) |
| HandOccNet (HO3D base) | 13.88 mm Aria | TBD (target <12 mm) |

- **PR-A (headline):** winning model + best FT recipe. Single Aria PA-MPJPE number.
- **PR-B (ablation):** frozen-backbone vs full-fine-tune on the same model. Proves the FT decision.

## 12. Appendix: canonical permutations

```python
# From /Users/stoic/Documents/Projects/vla/eval/phase0/phase0/config.py

ARIA_MPS_TO_OPENPOSE = {
    5: 0, 6: 2, 7: 3, 0: 4, 8: 5, 9: 6, 10: 7, 1: 8,
    11: 9, 12: 10, 13: 11, 2: 12, 14: 13, 15: 14, 16: 15, 3: 16,
    17: 17, 18: 18, 19: 19, 4: 20,
}
OP_VALID = sorted(ARIA_MPS_TO_OPENPOSE.values())  # [0, 2, 3, ..., 20], length 20

MANO_TO_OPENPOSE = [0, 13, 14, 15, 16, 1, 2, 3, 17, 4, 5, 6, 18, 10, 11, 12, 19, 7, 8, 9, 20]
```

Aria MPS native landmark convention (per projectaria_tools):
- 0-4: fingertips (thumb, index, middle, ring, pinky)
- 5: wrist
- 6-7: thumb intermediate + distal
- 8-10: index proximal/intermediate/distal
- 11-13: middle
- 14-16: ring
- 17-19: pinky
- 20: palm center (no OpenPose analog)

OpenPose 21-keypoint hand convention (used as canonical viz/eval frame):
- 0: wrist
- 1: thumb_cmc (DUMMY in MPS, excluded from OP_VALID)
- 2: thumb_mcp, 3: thumb_ip, 4: thumb_tip
- 5: index_mcp, 6: index_pip, 7: index_dip, 8: index_tip
- 9: middle_mcp, 10: middle_pip, 11: middle_dip, 12: middle_tip
- 13: ring_mcp, 14: ring_pip, 15: ring_dip, 16: ring_tip
- 17: pinky_mcp, 18: pinky_pip, 19: pinky_dip, 20: pinky_tip
